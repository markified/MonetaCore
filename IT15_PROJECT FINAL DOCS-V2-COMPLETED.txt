	API/ALGO/MODEL: 
	1. PayMongo API (Payment Gateway)
	   - Used for processing online card payments and GCash/e-wallet checkout sessions
	   - Integrated via PayMongoService (Services/PayMongoService.cs)
	   - Endpoints used: POST /v1/checkout_sessions, POST /v1/payment_intents
	   - Supports both direct card tokenization flow and hosted checkout redirect flow
	   - Simulation mode activates automatically when no secret key is configured

	2. RESTful API (Internal — MonetaCore Web API)
	   - Exposed under /api/ prefix, documented via Swagger/OpenAPI at /api/docs
	   - Endpoints: POST /api/invoices, PUT /api/invoices/{id}/status, POST /api/payments
	   - Used for external integrations and programmatic access
	   - Secured with cookie authentication and policy-based RBAC

	3. Revenue Calculation Algorithm (InvoiceCalculator)
	   - Custom algorithm in Services/InvoiceCalculator.cs
	   - Computes invoice Subtotal, TaxAmount (configurable rate), AdjustmentTotal, TotalAmount, BalanceDue
	   - BalanceDue = TotalAmount - AmountPaid +/- CreditNote/DebitMemo adjustments
	   - Recalculates automatically on payment or adjustment posting

	4. Profit & Loss Report Generation (ProfitLossPdfDocument)
	   - Uses QuestPDF library to generate structured PDF financial reports
	   - Aggregates revenue entries (invoiced amounts) vs. expense entries (adjustments/debit memos)
	   - Grouped and sorted by period (date range filter)

	Project Objectives:
	1. To develop a comprehensive invoice generation and tracking module that automates the creation, issuance, and monitoring of client invoices in real-time.
2. To implement a secure and efficient payment processing module that supports multiple payment methods, tracks payment statuses, and generates payment confirmations.
3. To design a credit and debit management module that accurately records financial adjustments, manages credit notes/debit memos, and maintains balanced account ledgers.
4. To build a revenue monitoring module that provides real-time financial dashboards, KPI alerts, and exportable profit/loss reports for finance managers.
5. To integrate an account integration module that enables data synchronization with external accounting systems via RESTful APIs, with full outbox event logging.


	Project Description:
	Target Audience / Users:
The system is designed for billing staff, accountants, finance managers, and system administrators in SMEs or service-based organizations requiring centralized billing and revenue tracking.

Potential Impact:
The application aims to reduce billing errors, accelerate payment cycles, improve financial visibility, and streamline account reconciliation for more transparent financial operations.

•  PDF Generation: QuestPDF
•  API Documentation: Swagger / Swashbuckle.AspNetCore 9.0.5
•  CI/CD: GitHub Actions
•  API Integration: PayMongo API (payment gateway), RESTful APIs for account integration

Why These Technologies:
Visual Studio Code and ASP.NET Core provide a robust, scalable environment for enterprise-grade web apps. MySQL ensures data integrity and is widely supported in PH hosting environments. PayMongo offers a local Philippine payment gateway with GCash and card support. Bootstrap 5 enables responsive UI design. Entity Framework Core simplifies database access with type-safe migrations. QuestPDF enables professional-grade PDF document generation without licensing fees.

	Prototype (Backend)
	SCREENSHOTS AND DESCRIPTION (Source Code)

	Backend Figure 1 — AppDbContext (Data/AppDbContext.cs)
	[Screenshot: AppDbContext.cs source code]
	Label: Entity Framework Core Database Context
	Description:
	AppDbContext is the central EF Core database context that maps all domain models to MySQL tables via Pomelo.EntityFrameworkCore.MySql. It defines DbSet properties for all entities: Invoices, InvoiceLineItems, Payments, Adjustments, AuditTrail, Clients, Users, AccountIntegrationEvents, and SystemConfiguration. Fluent API configurations enforce cascade delete rules (e.g., deleting an invoice cascades to its line items and adjustments) and restrict delete for audit and payment records to preserve financial history. The context is registered in Program.cs with the MySQL connection string resolved from appsettings.json / environment variables.

	Backend Figure 2 — InvoicesController.cs (Controllers/InvoicesController.cs)
	[Screenshot: InvoicesController.cs — Create and Cancel actions]
	Label: Invoice Generation Controller
	Description:
	The InvoicesController handles all invoice lifecycle operations: listing (with server-side pagination and filter), creating, viewing details, and cancelling. The Create action accepts a POST from the InvoiceCreateViewModel, validates the model state, auto-generates an invoice number via IInvoiceNumberService, calculates totals via InvoiceCalculator, saves to the database, and writes an audit entry via IAuditService. The Cancel action is gated behind the InvoiceCancellation RBAC policy. The controller is decorated with [RequireModule(SystemModule.InvoiceGeneration)] — if the module is disabled in System Configuration, all actions return a 403 Access Denied page automatically.

	Backend Figure 3 — PayMongoService.cs (Services/PayMongoService.cs)
	[Screenshot: PayMongoService.cs — CreateCheckoutSessionAsync method]
	Label: PayMongo API Integration Service
	Description:
	PayMongoService implements the IPayMongoService interface and encapsulates all HTTP communication with the PayMongo REST API. The CreateCheckoutSessionAsync method constructs a JSON payload with line item details, success/cancel redirect URLs, and the client's billing information. The request is authenticated via Basic Auth using the Base64-encoded secret key. The service resolves the secret key by first checking SystemConfiguration (database), then falling back to IOptions<PayMongoOptions> (user secrets / environment variable MONETACORE_PayMongo__SecretKey). If no key is found, the service returns a simulation result, enabling development without live credentials. API/Algo Used: PayMongo API v1 — POST /v1/checkout_sessions, POST /v1/payment_intents.

	Backend Figure 4 — AuditService.cs (Services/AuditService.cs)
	[Screenshot: AuditService.cs — LogAsync method]
	Label: Audit Trail Logging Service
	Description:
	The AuditService implements IAuditService and provides the LogAsync method called throughout all controllers on any state-changing operation. It receives the acting user's ID, username, the action name, the affected entity type and ID, and a metadata string (typically a JSON-serialized summary of the change). Each log entry is persisted to the AuditTrail table via AppDbContext. The service is registered as a scoped dependency in Program.cs. Audit entries are never deleted and have no cascade delete relationship — even if the related user or entity is removed.

	Backend Figure 5 — Program.cs (Program.cs — Security & Middleware Pipeline)
	[Screenshot: Program.cs — Authorization policies, middleware registration]
	Label: Application Startup & Security Pipeline
	Description:
	Program.cs configures the full ASP.NET Core middleware pipeline. Key security configurations include: (1) Policy-based RBAC using AddAuthorization() with named policies mapped in Security/AuthorizationPolicies.cs — each policy specifies the exact roles allowed (e.g., FinanceManagerOnly allows only FinanceManager; BillingOperations allows BillingStaff, Admin, SuperAdmin); (2) Cookie authentication with HttpOnly, Secure, and SameSite=Lax flags; (3) Anti-forgery token validation on all POST/PUT/DELETE forms; (4) API exception handling middleware (Middleware/ApiExceptionHandlingMiddleware.cs) that returns RFC 7807 ProblemDetails for /api/ routes; (5) /healthz endpoint for availability monitoring; (6) Swagger UI enabled only when ApiDocumentation:Enabled = true (disabled in production by default).

	Backend Figure 6 — RequireModuleAttribute.cs (Filters/RequireModuleAttribute.cs)
	[Screenshot: RequireModuleAttribute.cs]
	Label: Module Toggle Authorization Filter
	Description:
	RequireModuleAttribute is a custom IAsyncAuthorizationFilter that checks SystemConfigurationSettings at runtime to determine whether a given module is enabled. It is applied as [RequireModule(SystemModule.InvoiceGeneration)] on controller classes. If the targeted module's toggle is set to false in the System Configuration, the filter returns a 403 Forbidden result, preventing access to that controller's actions regardless of the user's role. This allows Super Admins to disable entire features (e.g., Payment Processing) without redeploying the application.


	API FUNCTIONS / FEATURES
	SCREENSHOTS AND DESCRIPTION (Source Code)

	API Figure 1 — Swagger / OpenAPI Documentation  (/api/docs)
	[Screenshot: Swagger UI at /api/docs showing available endpoints]
	Label: RESTful API Documentation via Swagger
	Description:
	MonetaCore exposes a RESTful API for programmatic and external system access, documented via Swagger/OpenAPI at the /api/docs endpoint (enabled via ApiDocumentation:Enabled configuration flag). The Swagger UI lists all available API endpoints with request/response schemas, HTTP methods, authentication requirements, and example payloads. This allows external accounting connectors and third-party clients to integrate with MonetaCore without needing to inspect source code. The API is versioned and grouped by module (Invoices, Payments, Integrations).

	How It Works:
	1. Swashbuckle.AspNetCore 9.0.5 scans all controllers decorated with [ApiController] and [Route("api/...")] attributes.
	2. XML comments and data annotations (e.g., [Required], [StringLength]) are reflected in the generated schema.
	3. The UI is served at /api/docs (Swagger JSON at /api/docs/swagger.json).
	4. Authentication is enforced: all API endpoints require valid session cookies and appropriate RBAC policy claims.
	5. The Swagger UI is conditionally disabled in production unless explicitly re-enabled via configuration.

	API Figure 2 — Invoices API (Controllers/Api/InvoicesApiController.cs)
	[Screenshot: InvoicesApiController.cs source code]
	Label: Invoice REST API Controller
	Description:
	The InvoicesApiController exposes HTTP endpoints for invoice operations to authorized external consumers:
	- POST   /api/invoices          — Create a new invoice (requires BillingOperations policy)
	- GET    /api/invoices/{id}     — Retrieve invoice details by ID
	- PUT    /api/invoices/{id}/status — Update invoice status (requires BillingOperations policy)

	How It Works:
	1. Requests are received as JSON via [FromBody] and validated against InvoiceApiModels request DTOs.
	2. The controller uses AppDbContext to persist changes and IInvoiceNumberService to auto-generate invoice numbers.
	3. Responses follow RFC 7807 — errors return ProblemDetails with HTTP status codes (400, 404, 409, 500).
	4. All write operations are logged to the audit trail via IAuditService.
	5. Anti-forgery is not required for API endpoints (JSON-only clients); CSRF protection relies on cookie SameSite policy.

	API Figure 3 — Payments API (Controllers/Api/PaymentsApiController.cs)
	[Screenshot: PaymentsApiController.cs source code]
	Label: Payment REST API Controller
	Description:
	The PaymentsApiController exposes:
	- POST /api/payments — Record a payment transaction against an invoice (requires BillingOperations policy)

	How It Works:
	1. The request body (PaymentApiModels.CreatePaymentRequest) is validated: InvoiceId must exist, Amount must be > 0, Method must be a valid DomainValues.PaymentMethod.
	2. If the method is PayMongo, the service delegates to IPayMongoService to initiate an API checkout session and returns the redirect URL in the response.
	3. On successful payment, the associated invoice's AmountPaid and BalanceDue are recalculated via InvoiceCalculator and saved.
	4. The transaction is persisted to paymenttransactions and an audit entry is written.
	5. Error cases (invoice not found, invoice already paid, zero balance) return appropriate ProblemDetails responses.

	API Figure 4 — PayMongo API Integration
	[Screenshot: PayMongoService.cs — HTTP request construction and response handling]
	Label: PayMongo Payment Gateway API
	Description:
	MonetaCore integrates with the PayMongo API (https://api.paymongo.com) to support online card and GCash payments from Philippine clients.

	How It Works:
	1. When a payment method of "PayMongo" is selected, PayMongoService.CreateCheckoutSessionAsync is called.
	2. The service builds a JSON body with line items, amounts (in centavos), currency (PHP), success_url, and cancel_url.
	3. An HTTP POST is sent to /v1/checkout_sessions with Basic Auth (Base64-encoded secret key).
	4. PayMongo returns a checkout session object with a hosted payment page URL.
	5. The user is redirected to the PayMongo-hosted page to complete payment securely.
	6. On return (success_url callback), the payment record is updated with the GatewayTransactionId and status is set to Completed.
	7. Invoice BalanceDue is updated accordingly.
	Simulation Mode: If no PayMongo secret key is configured, the service returns a simulated success result with a placeholder transaction ID — enabling full workflow testing without live credentials.

	API Figure 5 — Account Integration Outbox API
	[Screenshot: IntegrationsController or outbox API controller]
	Label: Account Integration Outbox Event API
	Description:
	The Account Integration module uses an outbox pattern to reliably dispatch billing events to external accounting systems.

	How It Works:
	1. When an invoice is created, paid, or cancelled, an AccountIntegrationEvent record is written to the database with Status = Pending.
	2. The Integrations controller dispatches Pending events to the configured external API (AccountingApiBaseUrl) using HttpClient.
	3. Authentication uses the configured mode (API Key via X-Api-Key header, or Bearer token).
	4. Successful dispatches update the event Status to Success and record the CorrelationId.
	5. Failed dispatches set Status to Failed and move the record to the Dead-Letter Queue.
	6. The Integrations page in the UI displays both the live event log and dead-letter queue, with retry capability for failed events.
	7. AutoSync can be enabled from System Configuration to automatically dispatch events on save.


	SECURITY FEATURES
	SCREENSHOTS AND DESCRIPTION (Source Code)

	Security Feature 1 — Authentication & Cookie Session Management
	[Screenshot: Program.cs — AddAuthentication / AddCookie configuration]
	Label: Cookie-Based Authentication with Secure Session Management
	Description:
	MonetaCore uses ASP.NET Core cookie authentication for stateful session management. Passwords are hashed using BCrypt (via PasswordService/IPasswordService) — plain-text passwords are never stored.

	How It Works:
	1. On login, the system validates the user's email and verifies the BCrypt password hash via IPasswordService.VerifyPassword().
	2. A claims-based authentication cookie is issued containing the user's ID, name, email, and role.
	3. Cookie flags: HttpOnly (prevents JavaScript access), Secure (HTTPS-only in production), SameSite=Lax (CSRF protection).
	4. Session timeout is configurable via SystemConfigurationSettings.SessionTimeoutMinutes (default: 480 minutes).
	5. Deactivated users (IsActive = false) are blocked at login — their cookie claims are rejected.
	6. Pending-approval users cannot log in until an Admin approves their registration (IsActive = false by default on self-registration).

	Security Feature 2 — Role-Based Access Control (Policy-Based RBAC)
	[Screenshot: Security/AuthorizationPolicies.cs and controller [Authorize] attributes]
	Label: Policy-Based Authorization (RBAC)
	Description:
	MonetaCore implements a two-layer RBAC system: role claims on the authentication cookie and named authorization policies evaluated at the controller level.

	How It Works:
	1. Security/AuthorizationPolicies.cs defines named policy constants: SuperAdminOnly, SuperOrMainAdmin, BillingOperations, FinanceOperations, FinanceManagerOnly, ClientManagement, AuditAccess, IntegrationsOperations, InvoiceCancellation.
	2. Each policy maps to one or more allowed roles (e.g., BillingOperations allows BillingStaff, Admin, SuperAdmin).
	3. Controllers are decorated with [Authorize(Policy = "...")] instead of hard-coded role strings.
	4. Any unauthorized access attempt returns a 403 Forbidden response and redirects to the Access Denied page.
	5. The Client role has enforced data isolation — clients can only see their own invoices and payments (query filters applied in controller actions).
	6. Super Admin has a "god mode" override — included in all policies.

	Security Feature 3 — Input Validation & Anti-Forgery (CSRF Protection)
	[Screenshot: ViewModel with [Required]/[StringLength] attributes, form with @Html.AntiForgeryToken()]
	Label: Server-Side Validation & CSRF Token Enforcement
	Description:
	All user input is validated both on the client side (Bootstrap validation UI) and server side (ASP.NET model binding + Data Annotations).

	How It Works:
	1. All ViewModels use Data Annotations ([Required], [StringLength], [EmailAddress], [Range]) to define validation rules.
	2. Controller actions check ModelState.IsValid before processing and return the form with validation errors if invalid.
	3. All state-changing form submissions include an AntiForgeryToken generated by @Html.AntiForgeryToken() in Razor views.
	4. The [ValidateAntiForgeryToken] attribute on all POST/PUT/DELETE actions rejects requests with missing or invalid tokens — preventing Cross-Site Request Forgery (CSRF) attacks.
	5. EF Core parameterized queries prevent SQL Injection — no raw SQL string concatenation is used anywhere.
	6. StringLength constraints on models map to database column length limits, preventing oversized input from reaching the database.

	Security Feature 4 — Secrets Management (No Hard-Coded Credentials)
	[Screenshot: appsettings.json showing empty PayMongo key, .env.example, and user secrets setup]
	Label: Secure Secrets Management via User Secrets & Environment Variables
	Description:
	No API keys, passwords, or connection strings with credentials are committed to source control.

	How It Works:
	1. The MonetaCore.csproj defines a UserSecretsId ("monetacore-dev-secrets") for local development secret storage outside the project directory.
	2. Sensitive values (PayMongo:SecretKey, database password) are stored in the dotnet user-secrets store during development.
	3. In production, values are injected via environment variables with the MONETACORE_ prefix (e.g., MONETACORE_PayMongo__SecretKey).
	4. appsettings.json only contains empty strings for sensitive fields — no real values are ever committed.
	5. .env.example documents all required environment variables without real values, serving as a setup reference.
	6. Production hosting should define these values directly in server environment variables or hosting control-panel settings, and they should never be committed to source control.

	Security Feature 5 — Audit Trail & Logging
	[Screenshot: AuditService.cs and Audit/Index.cshtml showing the audit log table]
	Label: Immutable Audit Trail with Full Action Logging
	Description:
	Every state-changing action in the system is recorded in an immutable audit trail.

	How It Works:
	1. IAuditService.LogAsync() is called in every controller action that creates, updates, or deletes data.
	2. Each audit entry records: acting user ID and name, action type, affected entity name and ID, metadata (JSON summary of changes), and UTC timestamp.
	3. The audittrail table has no cascade delete — audit records persist even if related invoices or users are removed.
	4. The Audit Trail view is accessible only to Auditors, Admins, and Super Admins (AuditAccess policy).
	5. Audit entries are never editable from the UI — the table is strictly read-only.
	6. Failed login attempts and access-denied events are also captured in the ASP.NET Core built-in logging pipeline (ILogger).

	Security Feature 6 — Global Error Handling Middleware & Health Check
	[Screenshot: Middleware/ApiExceptionHandlingMiddleware.cs and /healthz response]
	Label: API Exception Handling Middleware & Health Endpoint
	Description:
	MonetaCore includes a global error handling middleware for the API layer and a health check endpoint for deployment monitoring.

	How It Works:
	1. ApiExceptionHandlingMiddleware intercepts all unhandled exceptions for requests to /api/ paths or requests with Accept: application/json headers.
	2. In Development: returns a ProblemDetails response (RFC 7807) with the full exception message and stack trace for debugging.
	3. In Production: returns a generic "An unexpected error occurred" message with a traceId — no internal details are exposed to clients.
	4. Non-API routes fall through to ASP.NET Core's built-in error handling page (UseExceptionHandler("/Home/Error")).
	5. The /healthz endpoint returns HTTP 200 with "Healthy" — used by uptime monitoring tools and hosting health probes to verify the application is running.
	6. All API errors are logged via ILogger at Error level for server-side monitoring and diagnostics.
